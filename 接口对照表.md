### 32. 检查并颁发徽章
| 项目 | 内容 |
|------|------|
| 方法 | POST |
| 路径 | /api/badge/check |
| 认证 | Bearer Token |

**请求体：**
```json
{
  "trigger_type": "string (必填, 触发类型)",
  "record_type": "string (可选, trigger_type=record_create 时必填, 如 poop/sleep/water)",
  "invite_count": "number (可选, trigger_type=invite 时填当前邀请数)"
}
```

**trigger_type 可选值：**

| trigger_type | 触发场景 | 携带参数 |
|-------------|---------|---------|
| `record_create` | 用户创建一条记录后自动触发 | record_type |
| `invite` | 邀请好友成功后触发 | invite_count |
| `daily_check` | 每日定时任务检查（凌晨） | 无 |
| `login` | 用户登录时触发 | 无 |

**响应体：**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "badges_earned": [
      { "badge_id": "first_poop", "name": "初次体验", "icon": "🌟", "rarity": "common" },
      { "badge_id": "continuous_7", "name": "初露锋芒", "icon": "🔥", "rarity": "common" }
    ]
  }
}
```

> 返回本次新获得的徽章列表。前端收到后可弹出"获得新徽章"动画提示  
> **幂等性**：同一条件重复触发不会重复发放

| 状态 | 前端 | 后端 |
|------|------|------|
| 实现状态 | ✅ | ✅ |

## 数据源监听规则

> 更新日期：2026-05-25  
> 用途：从金山文档（KDocs）数据源定时抓取最新影视/电影/4K数据，自动同步到数据库  
> 驱动：DrissionPage 无头浏览器（Chromium headless）  
> 调度：APScheduler — 每个数据源独立 cron 配置（PC端可灵活设置）

---

### 设计架构

每个数据源独立配置，包含:
- **URL**：金山文档的分享链接
- **type**：同步到 `animes` 表时的 `type` 字段值
- **cron_expression**：该数据源独立的定时同步表达式（默认 `0 2,14 * * *`）
- **enabled**：是否启用定时抓取

PC 管理端通过接口 72~74（增删改查）管理数据源，每个源可单独修改 cron。

### 默认数据源

| 名称 | type | URL | 默认定时 |
|------|------|-----|---------|
| 影视剧（番剧） | `anime` | `https://www.kdocs.cn/l/co72a28MWkmI` | `*/15 * * * *`（每15分钟） |
| 电影 | `movie` | `https://www.kdocs.cn/l/cmbapmIwVsfi` | `0 2 * * *`（每天凌晨2点） |
| 4K资源 | `anime_4k` | `https://www.kdocs.cn/l/cdv0WUisFk3x?openfrom=docs` | `0 3 * * *`（每天凌晨3点） |

> 💡 每个源的 cron 可在 PC 管理端独立修改，以上为初始默认值。

---

### 执行流程

```
[定时触发(按各源cron)] → [从数据库读取该源配置] → [DrissionPage无头浏览器打开URL] 
    → [滚动加载全部内容] → [解析标题+链接] → [同步到数据库animes表]
                                              ↓
                                    [备份JSON到 date/ 目录]
```

**详细流程：**

1. **触发**：APScheduler 启动时从 `kdocs_sources` 表读取所有 `enabled=True` 的源，按各自的 `cron_expression` 注册独立定时任务（每个源一个 job）
2. **执行**：定时触发时，只抓取触发源的数据（不是全部源）
3. **打开页面**：启动 Headless Chromium（`--headless=new --no-sandbox --disable-gpu`），打开该源的 URL
4. **加载数据**：按 6000px 步进滚动页面，确保所有动态加载内容全部展开，每个分步等待 0.8s 保证渲染完成
5. **解析文本**：
   - 按行解析，识别标题行（包含 `1080P` / `720P` / `4K` 标记的行）
   - 识别百度网盘链接（匹配 `https://pan.baidu.com/s/`）
   - 识别夸克网盘链接（匹配 `https://pan.quark.cn/s/`）
   - 解析逻辑见 `app/services/kdocs_fetcher.py`
6. **同步入库**：以清理后的标题为指纹，做增量同步
   - 标题指纹相同且字段有变化 → UPDATE（仅变更字段：episode / status / urls / update_time）
   - 标题指纹不存在 → INSERT
7. **关闭浏览器**：每个数据源处理完后立即关闭 Chromium 实例
8. **本地备份**：每次抓取结果额外保存一份 JSON 到 `date/` 目录作为历史快照

---

### 标题解析规则

原始标题示例：`歌手 2026.1080P.更5.23`

| 输出字段 | 解析方式 | 示例结果 |
|---------|---------|---------|
| title | 去掉画质标记、更新标记、标签后的纯净标题 | `歌手 2026` |
| quality | 匹配 `4K` > `1080P` > `720P` | `1080P` |
| episode | 匹配 `更\d+\.?\d*[期集]?` 或 `\d+\.?\d*[期集]全` | `更5.23` |
| status | 包含"完结"→已完结 / "超前"→超前点播 / "铂金"→珍藏版 | `null` |
| update_time | 匹配 `\d{1,2}\.\d{1,2}` 格式，转为 `MM-DD` | `05-24` |
| baidu_url | 匹配 `https?://pan.baidu.com/s/[a-zA-Z0-9_-]+` | `https://pan.baidu.com/s/xxx` |
| baidu_password | 匹配 `?pwd=(\w+)` 或默认为 `1120` | `1120` |
| quark_url | 匹配 `https?://pan.quark.cn/s/[a-zA-Z0-9]+` | `https://pan.quark.cn/s/xxx` |

---

### 增量同步策略

```
新抓取条目 → 计算标题指纹(去空格/括号/特殊字符) 
         → 与数据库同 type 条目匹配
         ├─ 匹配成功 → 对比字段 → 有变化 → UPDATE episode/status/urls/update_time
         │                       → 无变化 → 跳过
         └─ 匹配失败 → INSERT 新行
```

**指纹算法：**
```python
# 小写 + 去掉括号内容 + 去除非中英文数字字符
title.lower().strip() → re.sub(r'[（(].*?[）)]', '', t) → re.sub(r'[^\w\u4e00-\u9fa5]', '', t)
```

---

### 代码结构

| 文件 | 功能 |
|------|------|
| `app/services/kdocs_fetcher.py` | 无头浏览器抓取，从数据库读取启用的数据源，解析页面内容 |
| `app/services/kdocs_sync_service.py` | 增量同步逻辑，匹配/更新/插入数据库 |
| `app/tasks/kdocs_tasks.py` | 定时任务封装，支持单个源同步和全量同步 |
| `app/tasks/scheduler.py` | APScheduler 定时调度，启动时动态注册各源独立 cron job |
| `date/*.json` | 每次抓取的本地快照备份 |

### 88. 上传公众号二维码
| 项目 | 内容 |
|------|------|
| 方法 | POST |
| 路径 | /api/admin/upload/qrcode |
| Content-Type | multipart/form-data |

**请求体：** `file` (File, 必填, png/jpg/jpeg/webp/gif, 限制 5MB)

**响应体：**
```json
{
  "code": 0,
  "data": { "url": "/static/qrcode/xxx.png" }
}
```

**错误码：** 40001 (类型不支持/文件过大)

> 前端使用流程：调用本接口上传图片→获得 `url`→填入 `PUT /api/admin/reserve` 的 `official_account.qrcode_url`
