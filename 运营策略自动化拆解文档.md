# 运营策略自动化拆解文档

> 目标：让大模型根据以下规则，自动执行用户触达、奖励下发、活动管理
> 运行环境：后端服务 + 定时任务 + 事件触发器

---

## 一、核心数据结构

### 1.1 用户表字段

| 字段                   | 类型     | 说明                 |
| ---------------------- | -------- | -------------------- |
| user_id                | string   | 用户唯一标识         |
| register_at            | datetime | 注册时间             |
| invite_count           | int      | 当前邀请人数         |
| last_invite_at         | datetime | 最后一次邀请成功时间 |
| continuous_record_days | int      | 连续记录天数         |
| last_record_at         | datetime | 最后一次记录时间     |
| badge_list             | array    | 已获得徽章ID列表     |
| couple_id              | string   | 绑定的情侣ID（可选） |
| family_id              | string   | 家族ID（可选）       |
| guardian_list          | array    | 守护者用户ID列表     |
| unlocked_features      | array    | 已解锁功能列表       |

### 1.2 事件日志表

| 字段       | 类型     | 说明                   |
| ---------- | -------- | ---------------------- |
| event_id   | string   | 事件唯一标识           |
| user_id    | string   | 触发用户               |
| event_type | string   | 事件类型（见下方枚举） |
| event_time | datetime | 触发时间               |
| event_data | json     | 事件附带数据           |

**事件类型枚举：**
- `record_create` - 用户完成记录
- `invite_success` - 邀请成功（被邀请人注册）
- `badge_earned` - 获得徽章
- `guardian_create` - 建立守护关系
- `couple_bind` - 绑定情侣
- `family_join` - 加入家族
- `continuous_days_milestone` - 连续记录里程碑

### 1.3 奖励记录表

| 字段         | 类型     | 说明                                           |
| ------------ | -------- | ---------------------------------------------- |
| reward_id    | string   | 奖励唯一标识                                   |
| user_id      | string   | 获得用户                                       |
| reward_type  | string   | 奖励类型（badge/feature/skin/coupon/physical） |
| reward_value | string   | 奖励值（如徽章ID、功能ID）                     |
| grant_at     | datetime | 下发时间                                       |
| grant_reason | string   | 触发原因（如invite_3人）                       |

## 二、用户分群规则

### 2.1 分群维度

| 分群维度 | 分群值     | 判断逻辑                            |
| -------- | ---------- | ----------------------------------- |
| 活跃度   | 高活跃     | 最近7天有≥5天记录                   |
| 活跃度   | 中活跃     | 最近7天有2-4天记录                  |
| 活跃度   | 低活跃     | 最近7天有0-1天记录                  |
| 活跃度   | 流失       | 连续14天无记录                      |
| 裂变能力 | 裂变者     | invite_count ≥ 5                    |
| 裂变能力 | 潜在线索   | 有分享行为但invite_count=0          |
| 裂变能力 | 普通用户   | 从未分享                            |
| 关系状态 | 有情侣     | couple_id不为空                     |
| 关系状态 | 有守护者   | guardian_list长度≥1                 |
| 关系状态 | 有家族     | family_id不为空且成员≥2             |
| 功能解锁 | 追番已解锁 | unlocked_features包含"anime_remind" |
| 功能解锁 | 追番未解锁 | invite_count < 8                    |
| 功能解锁 | 网盘已解锁 | invite_count ≥ 30                   |
| 功能解锁 | 网盘未解锁 | invite_count < 30                   |
| 连续记录 | 7天达成    | continuous_record_days ≥ 7          |
| 连续记录 | 14天达成   | continuous_record_days ≥ 14         |
| 连续记录 | 30天达成   | continuous_record_days ≥ 30         |

### 2.2 分群更新频率

| 分群类型 | 更新频率   | 说明                     |
| -------- | ---------- | ------------------------ |
| 活跃度   | 每日凌晨   | 基于前7天数据计算        |
| 裂变能力 | 实时       | invite_count变化时更新   |
| 关系状态 | 实时       | 绑定/解绑时更新          |
| 功能解锁 | 实时       | invite_count跨阈值时更新 |
| 连续记录 | 每次记录后 | 连续天数变化时更新       |

## 三、触发条件与推送规则

### 3.1 事件触发类（实时）

| 触发事件       | 条件                          | 推送对象 | 推送渠道                | 推送内容模板ID      |
| -------------- | ----------------------------- | -------- | ----------------------- | ------------------- |
| 完成首次记录   | 用户记录数=1                  | 本人     | 小程序内弹窗            | msg_welcome_01      |
| 邀请成功       | 用户A的invite_count从N变为N+1 | 用户A    | 小程序内弹窗            | msg_invite_progress |
| 跨解锁阈值     | invite_count达到1/3/8/15/30   | 用户A    | 小程序内弹窗 + 服务通知 | msg_feature_unlock  |
| 获得徽章       | 用户获得新徽章                | 本人     | 小程序内弹窗            | msg_badge_earned    |
| 连续记录7天    | continuous_record_days=7      | 本人     | 小程序内弹窗            | msg_continuous_7    |
| 连续记录14天   | continuous_record_days=14     | 本人     | 小程序内弹窗            | msg_continuous_14   |
| 连续记录30天   | continuous_record_days=30     | 本人     | 小程序内弹窗 + 徽章     | msg_continuous_30   |
| 情侣绑定成功   | couple_id从空变为有值         | 双方     | 小程序内弹窗            | msg_couple_bound    |
| 守护者授权成功 | 新建立守护关系                | 双方     | 小程序内弹窗            | msg_guardian_bound  |

### 3.2 定时触发类（每日/每周）

| 任务名称       | 触发时间    | 目标用户                     | 条件                | 推送内容                           |
| -------------- | ----------- | ---------------------------- | ------------------- | ---------------------------------- |
| 睡前提醒       | 每日21:00   | 有睡眠记录但今日未记录的用户 | 今日record_count=0  | 「今天还没记录睡眠，睡前来一记？」 |
| 姨妈预测提醒   | 每日08:00   | 有姨妈记录且预测3天内来潮    | 周期预测算法        | 「姨妈预计X天后到，提前准备～」    |
| 记录打卡提醒   | 每日20:00   | 今日未做任何记录的用户       | 今日record_count=0  | 「今天还没记录，点一下只需要2秒」  |
| 邀请进度提醒   | 每周一10:00 | invite_count在1-29之间的用户 | 距离上次邀请>7天    | 「你还差X人就能解锁网盘资源」      |
| 周报推送       | 每周日20:00 | 本周有≥3天记录的用户         | 周记录天数≥3        | 「你的本周健康周报已生成」         |
| 流失召回       | 每日10:00   | 连续7天未记录的用户          | 距离上次记录=7-10天 | 「好久不见，你的健康日历还等着你」 |
| 流失预警       | 每日10:00   | 连续5天未记录的用户          | 距离上次记录=5-6天  | 「连续5天没记录了，还好吗？」      |
| 裂变排行榜更新 | 每周一00:00 | 全体用户                     | 无                  | 更新排行榜数据（不主动推送）       |

### 3.3 条件触发类（用户主动触发后的反馈）

| 用户行为        | 条件       | 触发内容                       |
| --------------- | ---------- | ------------------------------ |
| 点击未解锁的??? | 功能未解锁 | 弹出锁定提示弹窗，展示当前进度 |
| 查看邀请中心    | 无         | 实时刷新邀请人数和各阶梯状态   |
| 点击生成海报    | 任意       | 根据当前用户数据生成对应海报   |
| 查看排行榜      | 无         | 展示邀请人数Top 10用户         |

### 4.3 奖励下发流程

```
用户触发条件（如邀请成功）
    ↓
检查是否已获得该奖励（防重复）
    ↓
未获得 → 执行下发
    ↓
更新用户状态（unlocked_features / badge_list）
    ↓
记录奖励日志（reward_log表）
    ↓
触发推送通知（弹窗/服务通知）
```

### 4.4 组队类奖励特殊逻辑

```
三人成团活动：
1. 用户A创建队伍 → 生成队伍ID
2. 用户B、C通过分享链接加入 → 队伍成员数+1
3. 队伍满3人 → 开始72小时倒计时
4. 倒计时内，每人记录次数≥2 → 全员解锁奖励
5. 任意条件不满足 → 活动失败，无奖励

需要存储的数据结构：
team = {
  team_id,
  members: [userA, userB, userC],
  created_at,
  deadline,
  status: 'pending' | 'success' | 'failed',
  member_records: { userA: 2, userB: 1, userC: 0 }
}
```

## 五、自动运营流程示例

### 5.1 新用户注册流程

```
用户注册成功
    ↓
【实时】默认解锁基础功能（拉屎、姨妈）
    ↓
【实时】推送欢迎弹窗：「欢迎来到记录生活」
    ↓
【实时】检查是否有邀请人（URL参数）→ 若有，触发invite_success事件
    ↓
【次日08:00】推送引导：「试试记录今天的拉屎或姨妈」
```

### 5.2 用户接近解锁阈值流程

```
用户当前invite_count=14
    ↓
距离15人阈值还差1人
    ↓
【触发条件】距离上次邀请>3天
    ↓
【推送】「还差1个朋友，就能解锁追番预告功能」
    ↓
推送渠道：小程序内弹窗 + 服务通知
    ↓
用户邀请成功 → invite_count=15
    ↓
自动解锁追番预告
    ↓
弹窗通知：「恭喜！追番预告已解锁」
```

### 5.3 流失用户召回流程

```
用户连续7天无记录
    ↓
进入「低活跃」分群
    ↓
Day 7 10:00 推送：「连续7天没记录了，我们想你了」
    ↓
Day 10 10:00 推送：「你的健康日历还在等你，回来看看吧」
    ↓
Day 14 10:00 推送：「最后一次提醒，再不来我们就安静了」
    ↓
Day 15 进入「流失」分群，停止推送（成本控制）
    ↓
若用户回来记录 → 移出流失分群，推送欢迎回来弹窗
```

### 5.4 裂变排行榜自动结算

```
每周一 00:00 触发
    ↓
查询全量用户，按invite_count降序取Top10
    ↓
更新排行榜缓存
    ↓
周一 10:00 向Top10用户推送：「恭喜你进入邀请榜Top10，点击查看」
    ↓
月底结算：Top3用户 → 触发实物奖励下发
    ↓
记录奖励日志，通知运营人员发货
```

## 六、配置化要求（运营后台可配）

| 配置项         | 类型     | 默认值 | 说明                      |
| -------------- | -------- | ------ | ------------------------- |
| 解锁阈值       | 映射表   | 见4.1  | 人数→解锁功能，可动态调整 |
| 推送时间       | 时间配置 | 见3.2  | 各定时任务执行时间        |
| 推送开关       | 布尔值   | true   | 可单独关闭某类推送        |
| 流失定义天数   | 整数     | 14     | 用户流失判断标准          |
| 低活跃定义天数 | 整数     | 7      | 低活跃判断标准            |
| 排行榜人数     | 整数     | 10     | Top N展示                 |
| 活动有效期     | 日期范围 | 见日历 | 各活动起止时间            |
| 组队倒计时     | 整数     | 72     | 三人成团的小时数          |

## 七、防刷与边界处理

| 场景                     | 处理规则                    |
| ------------------------ | --------------------------- |
| 同一设备/微信号多次注册  | 仅第一次计入邀请，后续过滤  |
| 用户A邀请用户B，B又邀请A | 不计入循环邀请，防刷        |
| 用户自己邀请自己         | 过滤                        |
| 奖励重复下发             | 每次下发前检查是否已有      |
| 活动结束后触发条件       | 不执行奖励下发              |
| 推送频率限制             | 同一用户每天最多3条推送     |
| 夜间静默                 | 22:00-08:00不推送非紧急消息 |
